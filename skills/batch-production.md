# 批量生产模式

当用户要求连续生成多章时（如"今天写 5 章"或"把第二卷写完"），采用此工作流。

# 执行流程

## 1. 预检

```
project_get(project_id) → 确认项目状态
batch_status(project_id) → 确认当前进度
outline_get(project_id) → 确认待写章节有大纲
```

预检通过条件：
- 项目 needHuman 为 false
- 待写章节在大纲中有定位
- open_loops 数量 < 12（过多未回收伏笔说明需要先整理）

## 2. 逐章生成

对每一章执行完整的 chapter-workflow（参考 chapter-workflow.md skill），但做以下优化：

### 批量效率优化
- 连续写时，context_get_state 可以使用上一章更新后的缓存
- 如果相邻章节场景连贯，减少重复上下文读取
- 但绝不跳过 chapter_update_summary —— 这是连续性的命脉

### 批量质量阈值
- 如果某章 QC 不通过，尝试自动修复一次
- 如果修复后仍不通过，暂停并报告
- 不要为了凑数量而放过低质章节

## 3. 每 3 章检查点

每写 3 章后做一次综合检查：
- 故事是否按大纲预期方向推进？
- 节奏是否合适？（有没有拖沓或太赶）
- 角色是否有足够的发展？
- open_loops 是否健康？

如果偏离大纲超过预期，暂停并向用户报告。

## 4. 完成报告

批量完成后输出：
```
📊 批量生成报告
━━━━━━━━━━━━━━
生成章节：第 X 章 ~ 第 Y 章（共 N 章）
总字数：XX,XXX 字
平均字数：X,XXX 字/章
QC 通过率：X/N

📖 剧情推进：
- [概括本批次主要剧情进展]

🪝 章末钩子：
- 第 X 章：[钩子概括]
- 第 Y 章：[钩子概括]

⚠️ 注意事项：
- [如有一致性风险或需要调整的地方]

📋 下一步建议：
- [建议接下来应该做什么]
```

# 异常处理

| 异常 | 处理 |
|------|------|
| 大纲不足 | 暂停，建议先补充大纲 |
| QC 连续失败 | 暂停，可能需要调整方向 |
| 伏笔溢出 | 建议先回收部分伏笔 |
| 字数持续不达标 | 检查是否场景设计不足 |
| needHuman 被触发 | 立即停止，报告原因 |

# 性能目标

- 每章生成时间目标：< 2 分钟（取决于 AI 服务响应）
- 一致性错误率目标：< 5%
- QC 首次通过率目标：> 80%
